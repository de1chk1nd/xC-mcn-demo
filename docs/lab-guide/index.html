xC MCN Demo Lab Guide

Welcome to the F5 Distributed Cloud Multi-Cloud Networking lab. This guide provides step-by-step instructions for setting up the lab environment and running each use case.

Lab Architecture

Two AWS regions (eu-central-1, eu-west-1) with CE nodes, BIG-IP appliances, and Ubuntu web servers connected via Transit Gateway.

Use the sidebar to navigate through setup and each use case. A full overview is available under Use Cases.

Basic Setup

Three steps: install OS tools, clone the repository, then set up a Python environment.

1. Install Tools

Linux (Ubuntu / Debian)

# Terraform
wget -O - https://apt.releases.hashicorp.com/gpg | sudo gpg --dearmor -o /usr/share/keyrings/hashicorp-archive-keyring.gpg
echo "deb [signed-by=/usr/share/keyrings/hashicorp-archive-keyring.gpg] https://apt.releases.hashicorp.com $(lsb_release -cs) main" | sudo tee /etc/apt/sources.list.d/hashicorp.list
sudo apt update && sudo apt install -y terraform

# Python 3 + pip
sudo apt install -y python3 python3-pip

# Tools
sudo apt install -y yq curl openssl git gettext-base

macOS

brew install terraform python yq curl openssl git gettext
Note gettext-base (Linux) / gettext (macOS) provides envsubst which is required by the use-case scripts.

2. Python Packages

Python dependencies are managed via requirements.txt in a virtual environment. This works the same on Linux and macOS.

Clone the repository

git clone https://github.com/de1chk1nd/xC-mcn-demo.git
cd xC-mcn-demo

Create venv and install dependencies

python3 -m venv .venv
source .venv/bin/activate
pip install -r requirements.txt

Lab Setup

This assumes you completed the Basic Setup (tools installed, repo cloned). If you are returning on a different day, start with Step 1 to re-activate your environment.

1. Activate Environment

Navigate into the repository and activate the Python virtual environment.

cd xC-mcn-demo
source .venv/bin/activate

2. Gather Credentials

You need two sets of credentials before configuring the lab.

F5 xC API Certificate

  1. Log in to the F5 Distributed Cloud Console.
  2. Navigate to Administration → Credentials → API Credentials.
  3. Create or download an API certificate (.p12 file).
  4. Note the password you set during creation.
  5. Copy the file into the repository:
cp ~/Downloads/your-cert.p12 ./setup-init/.xC/

AWS Credentials

Obtain temporary (STS) or static AWS credentials from your AWS account or lab portal:

  • aws_access_key_id
  • aws_secret_access_key
  • aws_session_token (if using STS / temporary credentials)

3. Configure

Copy the template and open it in your editor:

cp ./setup-init/template/config.yaml ./setup-init/config.yaml
vim ./setup-init/config.yaml

The configuration file has four sections:

4. Deploy

./setup-init/bin/initialize.sh init

This will automatically:

  • Detect your public IP
  • Generate a Certificate Authority in ./setup-init/.cert/ca/
  • Update AWS credentials in ~/.aws/credentials
  • Convert the xC P12 certificate to PEM
  • Run Terraform (fmt, init, plan, apply)

5. Post Install

Add local /etc/hosts entries and open SSH sessions to all lab servers.

Important Add local /etc/hosts entries before accessing any devices.

Linux

Hostfile

terraform -chdir="./infrastructure" output -raw etc-hosts | xclip -sel clip
sudo vim /etc/hosts

SSH Access

chmod +x ./setup-init/.ssh/ssh-key-permission_lnx.sh
./setup-init/.ssh/ssh-key-permission_lnx.sh all

macOS

Hostfile

terraform -chdir="./infrastructure" output -raw etc-hosts | pbcopy
sudo vim /etc/hosts

SSH Access

chmod +x ./setup-init/.ssh/ssh-key-permission_mac.sh
./setup-init/.ssh/ssh-key-permission_mac.sh all

Lab Deletion

When you are done with the lab, tear down all infrastructure and clean up local artifacts.

1. Update Credentials (if expired)

If your AWS credentials have expired since the initial setup, update config.yaml and run:

./setup-init/bin/initialize.sh update-creds

2. Destroy Infrastructure

This will destroy all AWS resources, remove generated certificates, clean up the AWS profile, and reset auto-populated config values.

./setup-init/bin/delete.sh

3. Remove /etc/hosts Entries

Manually remove the lab entries you added during post install:

sudo vim /etc/hosts
What the delete script removes
  • All Terraform-managed AWS infrastructure
  • Generated xC PEM certificate (.xC/xc-curl.crt.pem)
  • CA and domain certificates (.cert/)
  • AWS credentials profile (xc-mcn-lab) from ~/.aws/credentials
  • Auto-populated values in config.yaml (IP, CA paths)

North-South: CE via Cloud Load Balancer

Create HTTPS load balancers that are advertised directly on the Customer Edge (CE) rather than on the Regional Edge. Traffic never leaves the local region and does not traverse the global F5 network.

An AWS Network Load Balancer (NLB) in each region forwards external traffic to the CE node. A default Web Application Firewall policy is attached, providing SaaS-managed WAAP at the edge without requiring a Regional Edge.

Deploy

The script generates server certificates, uploads them to xC, and creates one HTTPS load balancer per region, each advertised on the CE virtual site in its respective region.

chmod +x "./xC-use-cases/North-South Loadbalancer - CE via CLB/bin/setup.sh"
"./xC-use-cases/North-South Loadbalancer - CE via CLB/bin/setup.sh"

Test — External (via NLB)

External clients reach the application through the AWS NLB, which forwards traffic to the CE node. The CE terminates the TLS session and applies the WAF policy locally. Requires local /etc/hosts entries (see Post Install).

→ app-ce-central.__DOMAIN__ — eu-central

→ app-ce-west.__DOMAIN__ — eu-west

To verify the WAF is active, append a simple XSS payload to the URL — the request should be blocked:

https://app-ce-central.__DOMAIN__?a=<script>

Test — Internal (via SSH)

Because the load balancer is advertised on the CE node itself (not on the Regional Edge), any host inside the same VPC can reach it directly by resolving the FQDN to the CE’s local IP address. This means the same WAF policy protects both external and internal traffic without any additional configuration.

This is a key advantage of CE-based load balancers: a single deployment covers two scenarios at once:

  • External → CE: Internet clients reach the app via the NLB, which NATs to the CE address. The CE terminates TLS, applies the WAF, and forwards to the origin.
  • Internal → CE: Local servers (e.g. Ubuntu web servers) resolve the same FQDN to the CE directly. Every internal request also passes through the xC WAF — no separate internal load balancer or policy is needed.
Why does this work? When a load balancer is advertised on a CE virtual site, the CE node binds the FQDN to its own local interfaces (inside and outside). Any DNS query within the VPC resolves to the CE’s private IP, so local traffic flows directly to the CE without leaving the network — but still receives the full xC WAF treatment.

SSH into one of the Ubuntu servers and test. If you have not imported the lab CA, you can skip TLS verification with -k:

# Normal request (skip TLS verification)
curl --silent -k https://app-ce-central.__DOMAIN__ | grep "Server address"

# XSS test -- WAF should block
curl --silent -k "https://app-ce-central.__DOMAIN__?a=<script>"

East-West: CE to CE

Create internal HTTPS load balancers for cross-region east-west traffic between Customer Edge sites. Each region gets an LB that routes to the opposite region’s origin pool.

No VPC peering or Transit Gateway is required — connectivity is provided entirely by the xC fabric. Both load balancers share the same FQDN (remote-web) but are advertised on different CE virtual sites, so each region resolves the name to its local CE and gets routed to the remote origin. A default Web Application Firewall policy is attached.

Deploy

The script generates a server certificate for the shared FQDN, uploads it to xC, and creates two HTTPS load balancers (one per region, each pointing to the opposite region’s origin pool).

chmod +x "./xC-use-cases/East-West Loadbalancer - CE to CE/bin/setup.sh"
"./xC-use-cases/East-West Loadbalancer - CE to CE/bin/setup.sh"

Test

SSH into an Ubuntu server and curl the internal LB. The response should come from the opposite region, proving cross-region connectivity via the xC fabric:

# From eu-central: response should come from eu-west
curl -k --silent https://remote-web.__DOMAIN__ | grep "Server name"

# From eu-west: response should come from eu-central
curl -k --silent https://remote-web.__DOMAIN__ | grep "Server name"

# WAF test -- should be blocked
curl -k --silent "https://remote-web.__DOMAIN__?a=<script>"
SSH intoLB NameTraffic FlowResponse from
EU-Central-1 lb-api-int-central Central ➔ West EU-West-1 NGINX
EU-West-1 lb-api-int-west West ➔ Central EU-Central-1 NGINX
How does this work? Both load balancers use the same FQDN remote-web.__DOMAIN__, but each is advertised only on the CE virtual site in its own region (SITE_NETWORK_INSIDE). When an Ubuntu server in eu-central resolves the FQDN, it reaches the local CE, which routes the request through the xC fabric to the origin pool in eu-west — and vice versa. This enables secure cross-region connectivity without any direct VPC peering.
