Architecture Overview

What is F5 Distributed Cloud?

F5 Distributed Cloud (xC) is a platform that helps organizations make their applications faster, more secure, and easier to manage — no matter where they run.

Whether your application is hosted in the public cloud, in a private data center, or spread across multiple environments: xC connects everything through one global platform with a single console and a unified policy engine. Instead of juggling separate tools for load balancing, WAF, DDoS, and networking, everything is consolidated in one place.

The result: faster deployments, consistent security everywhere, and significantly less operational overhead.

The Global Network (PoPs)

At the core of xC is a global network of Points of Presence (PoPs) — think of them as secure entry points distributed around the world.

When a user opens an application:

  • They automatically connect to the closest PoP
  • Traffic is inspected, secured, and optimized right there
  • Then it is safely forwarded to the application backend

Every PoP is a full Regional Edge (RE) — not just a cache node. It can terminate TLS, apply WAF policies, mitigate DDoS attacks, and route requests to any backend, regardless of whether it runs in AWS, Azure, GCP, or on-premises.

Because the PoPs are interconnected via a private high-performance backbone (not the public internet), traffic between locations is faster and more secure than traditional architectures.

What makes this powerful:

  • Users get faster response times — traffic is handled at the nearest edge
  • Applications are protected before traffic reaches them — attacks are stopped at the PoP
  • Organizations don’t need to build their own global infrastructure

Source: f5.com/cloud

Business Use Cases

Fast and Secure Public Applications

Deliver websites and APIs globally without worrying about infrastructure or attacks. The F5 network handles load balancing, security, and TLS — your team focuses on the application.

Business value: better user experience, higher conversion rates, fewer outages.

Secure Access to Private Applications

Employees and partners can access internal applications securely — without complex VPN setups. A lightweight Customer Edge in your environment connects to the platform automatically.

Business value: improved security (Zero Trust approach), easier access for users, reduced operational complexity.

Simplifying Multi-Cloud and Hybrid Environments

Applications often run in different places — AWS, Azure, on-prem, or all of the above. xC connects them into one consistent platform with unified policies and networking.

Business value: less complexity, faster deployments, more flexibility.

Connecting Legacy and Modern Applications

Existing systems in data centers can be securely connected or exposed without major redesign. Modernize at your own pace while keeping existing investments protected.

Business value: protect existing investments, enable gradual modernization, reduce risk during transformation.

Three Ways to Deploy

The platform is highly flexible and adapts to your environment — not the other way around. Depending on your requirements, you can deploy in three models:

ModelBest ForHow It Works
SaaS Quick wins, public-facing apps Traffic enters and exits via the F5 Global Network. No infrastructure to manage.
Hybrid Private backends, compliance Traffic enters via the Global Network but exits through a Customer Edge (CE) into your private environment.
Multi-Site East-west, edge computing Customer Edges at multiple locations connect through the xC fabric — secure cross-site communication without VPN tunnels.

The beauty of this approach: you can start with SaaS today and extend to Hybrid or Multi-Site tomorrow. Same console, same policies, no re-architecture required.

Putting It All Together

The following diagram shows how these three models work in practice:

  1. SaaS: Users connect directly through the F5 PoP network to applications via the internet — security and load balancing are applied at the edge, no infrastructure needed.
  2. Hybrid: Users connect via the PoP, which routes traffic through a private, encrypted tunnel to backend applications in your environment — no public exposure of your servers required.
  3. Multi-Site: Internal users and applications at any site can connect n:n with other sites — with full security capabilities (WAF, DDoS, access control) applied at every hop.

All three models are managed from one console, with centralized policies and centralized logging — regardless of how many sites, clouds, or deployment models you combine.

SaaS Model

The Fastest Path to Secure Application Delivery

In the SaaS model, your application traffic flows entirely through the F5 Global Network. Users connect to the nearest Regional Edge (PoP), where the platform terminates TLS, applies security policies (WAF, Bot Defense, DDoS), and forwards the request to your origin server over the internet.

This is the architecture most people know from traditional CDN and WAF providers — and it works the same way here. The difference: it’s just one deployment model out of three, and it shares the same policy engine and console as the more advanced options.

When to Use This

  • Public-facing web applications and APIs
  • You want WAF, DDoS, and Bot protection without deploying any infrastructure
  • Your origin server is reachable over the internet (e.g. behind an AWS NLB, Azure LB, or a public IP)
  • You need a quick, proven setup — minutes, not weeks

What You Get

  • Global load balancing with anycast across 30+ PoPs
  • Web Application Firewall with managed rulesets (OWASP Top 10, CVEs, custom rules)
  • DDoS mitigation at the edge, before traffic reaches your infrastructure
  • TLS management with automated or custom certificates
  • Observability — request logs, security events, and dashboards out of the box
In this lab The RE Only use case demonstrates the SaaS model: traffic enters via the Regional Edge and exits directly to the AWS backend via the internet.

Hybrid Model

Extending the Platform Into Your Private Environment

The SaaS model works great for public backends — but what if your application runs in a private subnet, behind a firewall, or in an on-premises data center? That’s where the Customer Edge (CE) comes in.

A Customer Edge is a lightweight node (virtual machine or bare metal) that you deploy in your own environment. It connects back to the F5 Global Network automatically — no inbound firewall rules, no VPN configuration, no public IPs required. Traffic enters via the Regional Edge (just like SaaS), but instead of exiting over the internet, it travels through the private xC fabric to the CE, which forwards it to your local backend.

Why This Matters

Traditional hybrid architectures require you to set up VPN tunnels, manage routing tables, and deploy separate security appliances at each location. With xC, you get all of that in a single deployment:

  • One console for security policies across cloud and on-prem
  • No inbound firewall rules — the CE initiates the connection outward
  • Consistent WAF/Bot/DDoS protection regardless of where the backend runs
  • Faster time-to-service — deploy a CE, point the load balancer to it, done

You consolidate load balancing, security, and networking into one platform. That’s less infrastructure to manage, fewer moving parts, and faster deployments.

When to Use This

  • Applications in private subnets (AWS VPC, Azure VNet, on-prem DC)
  • You want the same security posture for cloud and on-prem workloads
  • Compliance requirements that prevent direct internet exposure of backends
  • You’re looking to consolidate multiple tools (LB + WAF + networking) into one platform
In this lab The RE to CE and RE to CE via BIG-IP use cases demonstrate the Hybrid model: traffic enters via RE and exits through the CE to private backends.

Multi-Site Model

Your Private PoPs — Everywhere

Think of the Customer Edge as more than just a gateway. In the Multi-Site model, the CE becomes a full application proxy at each location — capable of running load balancers, web application firewalls, and even container workloads directly at the edge.

In simple terms: you extend the F5 Global Network with your own private PoPs. Each CE site can terminate traffic locally, apply security policies, and communicate with other CE sites through the xC fabric — without VPN tunnels, without VPC peering, without complex routing tables.

East-West Without the Complexity

Traditionally, connecting applications across regions or data centers means setting up VPN tunnels, Transit Gateways, or MPLS circuits. Each new connection adds complexity, cost, and latency.

With xC Multi-Site, it’s different: every CE automatically connects to every other CE through the xC fabric. Want to route traffic from Frankfurt to Dublin? Just create a load balancer that points from one CE to the other — the fabric handles the rest. Encrypted, fast, and with full security policies applied.

What a Customer Edge Can Do

  • Local TLS termination — no round-trip to the cloud for every request
  • Local WAF and security policies — same policies as the global platform, enforced at the edge
  • East-west load balancing — route traffic between sites with low latency
  • Service discovery — automatically discover Kubernetes services or other endpoints
  • Edge computing (vk8s) — run container workloads directly on the CE, managed from the xC console

When to Use This

  • Cross-region or cross-cloud communication (east-west traffic)
  • Applications that need local processing at the edge (latency-sensitive workloads)
  • You want a consistent security and networking layer across all your sites
  • You’re replacing or consolidating VPN/MPLS/Transit Gateway setups

Security Services

The Challenge

In the previous chapter, we explored how F5 Distributed Cloud connects applications across clouds, data centers, and edge locations. The flexibility is powerful — but it comes with a challenge: the more places you deploy, the more places you need to protect.

The reality for most organizations is not “all in the cloud” or “stay on-premises” — it’s hybrid. Applications live everywhere: in public clouds, in private data centers, at the edge. And attackers don’t care where your app runs — they look for the weakest link.

This is where F5 Distributed Cloud Security Services come in. Every security function can run on the Regional Edge (the global PoP network) or on the Customer Edge (your own sites) — or both. Same policies, same engine, same console. No matter where you deploy, your security posture stays consistent.

What’s Available

F5 Distributed Cloud offers a comprehensive set of security services — all SaaS-managed, all integrated into one platform. Here’s what’s in the toolbox:

WAF & Reverse Proxy

Every application needs a front door — and that front door should be smart. The F5 Web Application Firewall combines a full reverse proxy with AI-powered threat detection. It handles SSL/TLS termination, enforces mTLS, applies IP reputation scoring, rate limiting, and service policies — all in one layer.

What makes it different: the WAF engine is the same proven F5 technology trusted by the world’s largest enterprises, now delivered as SaaS with managed signature updates and zero-day protection via threat campaigns. No tuning, no lag — new attack patterns are blocked within hours, not days.

  • AI-assisted analysis — reduces false positives and adapts to your traffic patterns
  • Threat campaigns — proactive protection against emerging CVEs and zero-days
  • Consistent enforcement — same WAF policy on RE and CE, managed from one console

API Security

APIs are the backbone of modern applications — and the number one attack vector. F5 Distributed Cloud protects APIs you already know about and helps you find the ones you don’t.

Import your OpenAPI/Swagger specification, and the platform automatically validates every request against your schema — wrong parameters, unexpected methods, oversized payloads get blocked before they reach your backend. Add JWT validation to verify caller identity, and you have a complete API security layer.

  • Swagger/OpenAPI enforcement — automatic schema validation, block non-conforming requests
  • JWT validation — verify tokens at the edge, before traffic reaches your app
  • Positive security model — only allow what’s defined, deny everything else

API Discovery

You can’t protect what you don’t know exists. Many organizations have “shadow APIs” — endpoints that were deployed but never documented, or legacy APIs that everyone forgot about. These are the endpoints attackers find first.

F5 Distributed Cloud analyzes your live traffic and automatically discovers API endpoints, maps their parameters, and generates OpenAPI specifications. It then monitors for OWASP API Top 10 attacks — injection, broken authentication, excessive data exposure — even on APIs you didn’t know you had.

  • Automatic endpoint discovery — find every API, documented or not
  • Swagger export — generate OpenAPI specs from observed traffic
  • OWASP API Top 10 — continuous monitoring and protection across all discovered endpoints

Bot Protection

Not every visitor is human. Bots account for a significant portion of internet traffic — and while some are harmless (search crawlers), others steal credentials, scrape content, or abuse business logic.

F5 Distributed Cloud offers two levels of bot defense. Basic protection uses signature-based detection to block known bad bots. Advanced protection goes further — it analyzes client behavior in real-time using JavaScript signals, telemetry, and machine learning to distinguish real users from sophisticated bots. No CAPTCHAs. No image puzzles. Users don’t even notice it’s there.

  • Signature-based detection — block known bot frameworks and scrapers
  • Behavioral analysis — detect sophisticated bots that mimic human behavior
  • Invisible to users — no CAPTCHAs, no friction, no impact on user experience

Client-Side Defense & Mobile App Shield

Security doesn’t end at the server. Modern web applications load dozens of third-party scripts (analytics, ads, chat widgets) — any of which could be compromised. And mobile apps face a different threat: reverse engineering, tampering, and credential theft on the device itself.

F5 Client-Side Defense monitors all JavaScript running in the user’s browser and detects malicious modifications to third-party libraries — protecting against Magecart-style supply chain attacks. Mobile App Shield extends this to native apps, hardening them against reverse engineering and runtime tampering without requiring code changes.

  • 3rd-party script monitoring — detect compromised JavaScript libraries in real-time
  • Supply chain protection — block Magecart, formjacking, and skimming attacks
  • Mobile app hardening — protect native apps without changing application code

Why This Matters

Most security vendors offer pieces of the puzzle — a WAF here, a bot solution there, API security from a third vendor. Each tool has its own console, its own policy language, its own blind spots.

F5 Distributed Cloud brings all of this into one platform. One console. One policy engine. Deployed anywhere — on the global network or at your own sites. That’s not just convenience, it’s a fundamentally different security posture:

  • Consistent policies across every deployment model (SaaS, Hybrid, Multi-Site)
  • Correlated visibility — WAF events, bot activity, API anomalies, and client-side alerts in one dashboard
  • Faster response — when you see the full picture, you act faster

SectionParameterDescription

Choose it carefully — changing it later requires a full teardown and redeploy. Rules: lowercase letters, digits, and hyphens only ([a-z0-9-]), max 16 characters.

North-South: CE via Cloud Load Balancer

Create HTTPS load balancers that are advertised directly on the Customer Edge (CE) rather than on the Regional Edge. Traffic never leaves the local region and does not traverse the global F5 network.

An AWS Network Load Balancer (NLB) in each region forwards external traffic to the CE node. A default Web Application Firewall policy is attached, providing SaaS-managed WAAP at the edge without requiring a Regional Edge.

Deploy

The script generates server certificates, uploads them to xC, and creates one HTTPS load balancer per region, each advertised on the CE virtual site in its respective region.

chmod +x "./xC-use-cases/Architecture/CE-via-CLB/bin/setup.sh"
"./xC-use-cases/Architecture/CE-via-CLB/bin/setup.sh"

Test — External (via NLB)

External clients reach the application through the AWS NLB, which forwards traffic to the CE node. The CE terminates the TLS session and applies the WAF policy locally. Requires local /etc/hosts entries (see Post Install).

→ app-ce-eu-central-1.__DOMAIN__ Test WAF
→ app-ce-eu-west-1.__DOMAIN__ Test WAF

Test — Internal (via SSH)

Because the load balancer is advertised on the CE node itself (not on the Regional Edge), any host inside the same VPC can reach it directly by resolving the FQDN to the CE’s local IP address. This means the same WAF policy protects both external and internal traffic without any additional configuration.

This is a key advantage of CE-based load balancers: a single deployment covers two scenarios at once:

  • External → CE: Internet clients reach the app via the NLB, which NATs to the CE address. The CE terminates TLS, applies the WAF, and forwards to the origin.
  • Internal → CE: Local servers (e.g. Ubuntu web servers) resolve the same FQDN to the CE directly. Every internal request also passes through the xC WAF — no separate internal load balancer or policy is needed.
Why does this work? When a load balancer is advertised on a CE virtual site, the CE node binds the FQDN to its own local interfaces (inside and outside). Any DNS query within the VPC resolves to the CE’s private IP, so local traffic flows directly to the CE without leaving the network — but still receives the full xC WAF treatment.

Certificate-Based Access Control with Service Policies

After mTLS authentication, the load balancer injects the x-forwarded-client-cert (XFCC) header containing the client certificate details. You can use xC Service Policies to enforce access control based on these certificate fields.

The setup script creates a service policy sp-${STUDENT}-mtls-cert-check that demonstrates this pattern:

  • Rule 1 (Deny): If the XFCC header does not match CN=...@mordor.de → block the request (403)
  • Rule 2 (Allow): Everything else (i.e. @mordor.de users) → allow
Manual step required The service policy is created by the setup script, but must be manually assigned to the load balancer lb-mtls in the xC Console: HTTP Load Balancer → Edit → Common Security Controls → Service Policies.

The key insight is that Service Policies evaluate after the mTLS handshake and header injection. This means you can use standard header matching (regex, exact, invert) on the XFCC Subject field to implement fine-grained access control based on client certificate attributes like CN, OU, O, or email.

Regex Pattern

xC uses RE2 regex syntax. The pattern used in this lab:

CN=[^,]*@mordor\.de

This matches any XFCC header where the CN field ends with @mordor.de. Combined with invert_matcher: true, it denies all requests where this pattern does not match.
