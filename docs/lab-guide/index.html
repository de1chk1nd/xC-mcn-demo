Basic Setup

Three steps: install OS tools, clone the repository, then set up a Python environment.

1. Install Tools

Linux (Ubuntu / Debian)

# Terraform
wget -O - https://apt.releases.hashicorp.com/gpg | sudo gpg --dearmor -o /usr/share/keyrings/hashicorp-archive-keyring.gpg
echo "deb [signed-by=/usr/share/keyrings/hashicorp-archive-keyring.gpg] https://apt.releases.hashicorp.com $(lsb_release -cs) main" | sudo tee /etc/apt/sources.list.d/hashicorp.list
sudo apt update && sudo apt install -y terraform

# Python 3 + pip
sudo apt install -y python3 python3-pip

# Tools
sudo apt install -y yq curl openssl git gettext-base

macOS

brew install terraform python yq curl openssl git gettext
Note gettext-base (Linux) / gettext (macOS) provides envsubst which is required by the use-case scripts.

2. Python Packages

Python dependencies are managed via requirements.txt in a virtual environment. This works the same on Linux and macOS.

Clone the repository

git clone https://github.com/de1chk1nd/xC-mcn-demo.git
cd xC-mcn-demo

Create venv and install dependencies

python3 -m venv .venv
source .venv/bin/activate
pip install -r requirements.txt

Lab Setup

This assumes you completed the Basic Setup (tools installed, repo cloned). If you are returning on a different day, start with Step 1 to re-activate your environment.

1. Activate Environment

Navigate into the repository and activate the Python virtual environment.

cd xC-mcn-demo
source .venv/bin/activate

2. Gather Credentials

You need two sets of credentials before configuring the lab.

F5 xC API Certificate

  1. Log in to the F5 Distributed Cloud Console.
  2. Navigate to Administration → Credentials → API Credentials.
  3. Create or download an API certificate (.p12 file).
  4. Note the password you set during creation.
  5. Copy the file into the repository:
cp ~/Downloads/your-cert.p12 ./setup-init/.xC/

AWS Credentials

Obtain temporary (STS) or static AWS credentials from your AWS account or lab portal:

  • aws_access_key_id
  • aws_secret_access_key
  • aws_session_token (if using STS / temporary credentials)

3. Configure

Copy the template and open it in your editor:

cp ./setup-init/template/config.yaml ./setup-init/config.yaml
vim ./setup-init/config.yaml

The configuration file has four sections:

Student name rules Lowercase letters, digits, and hyphens only ([a-z0-9-]). Must start and end with a letter or digit. Maximum 16 characters. This value becomes part of your DNS domain suffix: __DOMAIN__.

4. Deploy

./setup-init/bin/initialize.sh init

This will automatically:

  • Detect your public IP
  • Generate a Certificate Authority in ./setup-init/.cert/ca/
  • Update AWS credentials in ~/.aws/credentials
  • Convert the xC P12 certificate to PEM
  • Run Terraform (fmt, init, plan, apply)

5. Post Install

Add local /etc/hosts entries and open SSH sessions:

# Copy /etc/hosts entries to clipboard
terraform -chdir="./infrastructure" output -raw etc-hosts | xclip -sel clip
sudo vim /etc/hosts

# Fix SSH key permissions and open sessions to all lab servers
./setup-init/.ssh/ssh-key-permission_lnx.sh all
Important Add local /etc/hosts entries before accessing any devices.

Lab Deletion

When you are done with the lab, tear down all infrastructure and clean up local artifacts.

1. Update Credentials (if expired)

If your AWS credentials have expired since the initial setup, update config.yaml and run:

./setup-init/bin/initialize.sh update-creds

2. Destroy Infrastructure

This will destroy all AWS resources, remove generated certificates, clean up the AWS profile, and reset auto-populated config values.

./setup-init/bin/delete.sh

3. Remove /etc/hosts Entries

Manually remove the lab entries you added during post install:

sudo vim /etc/hosts
What the delete script removes
  • All Terraform-managed AWS infrastructure
  • Generated xC PEM certificate (.xC/xc-curl.crt.pem)
  • CA and domain certificates (.cert/)
  • AWS credentials profile (xc-mcn-lab) from ~/.aws/credentials
  • Auto-populated values in config.yaml (IP, CA paths)

East-West: Network Connect (L3 VPN)

Beta This lab section is in beta. Routing only possible via GW01-to-GW01 CE nodes.

Establishes a full L3 VPN tunnel between both AWS sites using xC segments.

Steps

  1. Create a Segment: __STUDENT__-global-vrf
  2. Go to Multi-Cloud Network Connect → Site Management → Customer Edges → your CE → Manage
  3. On interface ens6, change VRF from SITE LOCAL INSIDE to Segment (Global VRF) and select the segment. Repeat for both GW01 (eu-west and eu-central).
  4. Check CE Routes
  5. Test access

Delete

Change interface setting back to SITE_LOCAL_INSIDE.
